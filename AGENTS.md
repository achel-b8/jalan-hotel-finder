# AGENTS.md

このファイルは、本リポジトリで実装を進める際の方針・ルールの一次情報です。
Codex/エージェントはこの内容を前提に作業してください。

## 参照優先度
1. `docs/CLI仕様書.md`
2. `docs/技術アーキテクチャ設計書.md`
3. `docs/じゃらん仕様調査.md`
4. `docs/実装TODOワークリスト.md`

## ドキュメント正本管理
- 正本（Source of Truth）は `AGENTS.md` と、参照優先度に列挙した4文書のみとする。
- 同種の草稿・旧版を追加する場合は、正本と同階層に重複配置しない。
- 旧版を保持する場合は `docs/archive/` に移動し、冒頭に「非正本」を明記する。
- 実装・レビュー時は、正本以外の文書を仕様根拠として参照しない。

## 実装方針
- 小さく実装し、小さくテストで保証する。
- タスクは可能な限り独立させ、依存タスク順を守る。
- 1タスク = 1PR/1コミットを推奨する。

## レイヤ依存ルール
- 依存方向は `domain -> application -> infrastructure -> cli/output` を基準とする。
- 逆方向依存を禁止する（例: `domain` から `infrastructure` を参照しない）。
- 層をまたぐデータ受け渡しは、呼び出し側都合ではなくドメイン都合で設計する。

## 共通ルール
- 各タスクで `pytest` テストを追加し、完了条件を満たすまでマージしない。
- ネットワーク依存テストは最小化し、基本はユニットテスト（fixture + mock）で保証する。
- t-wadaの推奨するTDD（Red -> Green -> Refactor）で進める。
- 仕様変更または機能追加を行った場合は、`pytest` 通過後に必ずCLIコマンドを実行し、意図した仕様どおりの出力・終了コードになっていることを確認する。
- CLI実行確認では、変更箇所に対応するコマンドと確認観点（例: 出力件数、並び順、終了コード）を明示する。
- 受け入れ要件に紐づくテスト名を付ける。
  - 特に `終了コード`、`重複排除`、`並列数制約` を明示する。

## タスク運用
- 着手前に `docs/実装TODOワークリスト.md` の依存関係を確認する。
- 完了時は、対応タスクのチェックを更新する。
- チェック更新の前に `pytest` がローカルで安定して通ることを確認する。

## 変更時のドキュメント同期
- 仕様に影響する実装変更では、同一コミットまたは同一PRで対応ドキュメントを更新する。
- 更新対象は `docs/CLI仕様書.md`、`docs/技術アーキテクチャ設計書.md`、`docs/じゃらん仕様調査.md`、`docs/実装TODOワークリスト.md` から該当箇所を選ぶ。
- ドキュメント更新が不要な場合は、PR説明またはコミットメッセージで不要理由を明記する。

## タスク完了チェックリスト
- 実装に対応するテストを追加または更新した。
- `python -m pytest` が `0 failed` で完走した。
- 仕様変更/機能追加時は、対象CLIコマンドを実行し、意図した仕様どおりの挙動を確認した。
- 変更を `push` した後、GitHub Actions `CI Tests` の成功を確認した。
- `docs/実装TODOワークリスト.md` の対象タスク状態を更新した。
- 仕様差分の有無を確認し、必要な `docs/*.md` を更新した。

## 完了判定（Definition of Done）
- 実装が受け入れ要件を満たす。
- 追加/更新したテストがすべて成功する。
- `python -m pytest` が `0 failed` で完走する。

## 初期ディレクトリ指針
- 実装コード: `src/jalan_hotel_finder/{cli,application,domain,infrastructure,output}`
- テストコード: `tests/` 配下で責務に対応した構成を維持する。

## 仕様衝突時の判断手順
- まず「参照優先度」に従って解釈し、優先度の高い文書を採用する。
- 優先度で解決できない場合は実装を止め、衝突箇所と解釈案を明示して確認を取る。
- 判断後は、衝突が再発しないよう該当ドキュメントを更新してから実装を再開する。
