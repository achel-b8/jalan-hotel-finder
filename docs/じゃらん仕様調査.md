# じゃらんnet 仕様調査メモ（CLI化向け）
最終確認日: 2026-02-24

## 1. 調査対象
- じゃらん検索結果（小エリア URL）
- じゃらんフリーワード検索（宿名検索）
- エリアコード定義（都道府県/大エリア/小エリア）

参照URL:
- https://www.jalan.net/jalan/doc/jws/data/area.html
- https://www.jalan.net/jalan/doc/jws/data/area.xml
- https://www.jalan.net/010000/LRG_010200/SML_010202/

## 2. URL規則
### 2.1 エリア検索（通常の宿一覧）
- 形式: `https://www.jalan.net/{都道府県6桁}/LRG_{大エリア6桁}/SML_{小エリア6桁}/`
- 例: `https://www.jalan.net/010000/LRG_010200/SML_010202/`

### 2.2 フリーワード検索（宿名検索）
- 形式: `https://www.jalan.net/uw/uwp2011/uww2011init.do?keyword=...`
- `keyword` の文字コードは固定仕様として断定しない（UTF-8/CP932 で挙動差が出るケースがある）。
- 2026-02-21 の観測では、日本語キーワードで「UTF-8 は0件、CP932 はヒット」のケースを確認したが、差分条件は未確定。
- `list` は `keyword` one-shot（1候補名=1リクエスト）で運用し、候補宿名は CP932 `%xx` でURLエンコードする。
- ページネーション（`dispStartIndex`）は追跡しない。多重ループを避け、候補名ごとの初回結果のみを照合対象にする。
- 将来拡張として `keyword` エンコード方式（`auto` / `utf8` / `cp932`）のCLI切替を保持する。

## 3. ページネーション
### 3.1 エリア検索
- 1ページ 30件。
- `idx` でオフセット指定（0, 30, 60...）。
- 画面上は `selectPage('30','2')` の JS 呼び出し。
- 例: `...?idx=30` で2ページ目を取得可能。

### 3.2 フリーワード検索
- `nav.pagerLink a[href]` に次ページURLが直接出る。
- 例: `.../uww2011search.do?...&dispStartIndex=30...`
- `dispStartIndex` がオフセット（0, 30, 60...）。
- `keyword=ホテル` 検索で `dispStartIndex=990` までのリンクを確認。
  - ただし「1000件上限」は推定値として扱う。
  - CLI実装の `list` は one-shot 方針のため、`dispStartIndex` を辿らない。

## 4. 実装で使える主要パラメータ
- `stayYear`, `stayMonth`, `stayDay`, `dateUndecided`
- `adultNum`, `roomCount`, `stayCount`
- `mealType`
  - `0`: 食事なし
  - `1`: 朝のみ
  - `2`: 夕のみ
  - `3`: 朝/夕あり
- `careKake`: 温泉掛け流し
- `careBathRent`: 貸切風呂・貸切露天
- `carePribateBath`: 露天風呂付き客室
- `careBath`: 内湯・大浴場

## 5. ユースケース実現可否
### 5.1 日付指定 + 都道府県（任意）+ こだわり条件
- 実現可能。
- 都道府県単位は `area.xml` で配下SMLに分解し、SML単位で総当たり取得して統合するのが安定。

### 5.2 宿名リストから宿泊可能宿を取得
- 実現可能。
- 方法A: フリーワード検索 (`uww2011`) を宿名ごとに実行。
- 方法B: エリア総当たり後に宿名部分一致（または候補URLの`/yadxxxxxx`一致）でローカル絞り込み。
- 実装難易度/実行時間の観点から、v1 CLIの `list` は方法Aの one-shot（1候補名=1取得）を採用する。

### 5.3 特定クーポンが使える宿
- 部分的に実現可能。
- 検索結果DOM上にクーポン情報（モーダル）自体は存在する。
- ただし「クーポン名をURLパラメータで直接指定して検索」する公開仕様は確認できず、DOM依存が強い。
- 初期実装は非対応とし、後続でDOM抽出を追加するのが現実的。

## 6. 追加要望への回答（内風呂/部屋風呂）
- 安定パラメータとして確認できたのは `carePribateBath=1`（露天風呂付き客室）。
- `careBath=1` は「内湯・大浴場」であり、客室風呂の保証にはならない。
- 一休のような「客室に内風呂あり」を直接保証する公開パラメータは確認できなかった。

## 7. ミニマル実装方針
- 先行実装対象:
  - 日付
  - 都道府県（任意。指定時はSML分解）
  - 人数（デフォルト1名）
  - 食事タイプ
  - 温泉掛け流し
  - 貸切風呂・貸切露天
  - 露天風呂付き客室（任意）
- 固定運用:
  - 失敗ポリシーは `stop`
  - 出力は標準出力 JSON
  - `list` は `keyword-one-shot` 固定
- 出力:
  - 宿名
  - 宿URL
  - プラン名
  - 価格
- ページング:
  - `idx` / `dispStartIndex` を辿って全件取得
- 重複排除:
  - 宿URLのパス（`/yadxxxxx/`）単位で統合

## 8. 構造化データ観測（2026-02-22）
- `https://www.jalan.net/010000/LRG_010200/SML_010202/` を取得し、`application/ld+json` を確認。
- 観測した `@type`:
  - `BreadcrumbList`
  - `FAQPage`
  - `Hotel`（複数件）
- `Hotel` オブジェクトに含まれる主な項目:
  - `name`
  - `url`（`/yadxxxxxx/`）
  - `priceRange`
  - `image`
  - `address`
- 備考:
  - `plan_name` は構造化データ上で安定的に確認できないため、DOM抽出を併用する。
  - 価格は `priceRange` が文字列のため、出力仕様（`price: int | null`）へ変換できない場合は `null` を許容する。

## 9. 待機時間・再試行の運用根拠（v1）
- エリア間待機 `100ms` は、アクセス制限回避と実行時間短縮の両立を狙った暫定値として採用する。
- 根拠:
  - じゃらん側の公開レート制限仕様が明示されていない。
  - v1は `stop` 固定であるため、保守的な初期値で制限発生リスクを抑える。
- 運用:
  - 制限兆候（`429/403`、制限キーワード）発生時は即停止し、待機時間・並列設定を再調整する。
  - 実測を蓄積し、将来版で待機時間を見直す。

## 10. 抽出フォールバック実装メモ（2026-02-22）
- `application/ld+json` から `@type=Hotel` を優先抽出し、`name` / `url` / `priceRange` を使用。
- `plan_name` は構造化データで安定しないためDOMフォールバックで補完。
- v1フォールバックで実装した代表セレクタ:

## 11. 追加観測（2026-02-24）
- 北海道の配下SML（42件）を同条件（`checkin=2026-03-10`, `mealType=3`, `careKake=1`）で確認したところ、`SML_013508` のみ `エラー画面`（「アクセスが集中」文言）を恒常的に返すことを確認。
