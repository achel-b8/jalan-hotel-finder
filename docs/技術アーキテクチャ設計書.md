# jalan-hotel-finder v1仕様ベース技術調査・アーキテクチャ設計

更新日: 2026-02-21  
対象資料:
- `docs/CLI仕様書.md`

## 1. 目的
- 実装前に、CLI仕様書に基づいて必要な言語・ライブラリ・ミドルウェアを確定する。
- 個人用CUIツールとして、過剰設計を避けた実行アーキテクチャを定義する。
- 対象は「よい温泉宿を短時間で横断比較したい個人利用者」とし、この利用目的に沿った既定値を採用する。

## 2. 要件サマリ（設計インプット）
### 2.1 機能要件
- じゃらん検索結果から次を抽出:
  - 宿名
  - 宿URL
  - プラン名
  - 価格
- 入力ユースケース:
  - 日付指定 + 都道府県 + こだわり条件
  - 宿名リスト検索
  - 特定クーポン検索（初期は難易度高）
- 重複排除:
  - 同一宿URLを統合
  - クエリ文字列を除去したパスベースで判定

### 2.2 非機能・運用要件
- Linux(WSL2) 上の Python 実行
- Playwright (Chromium) による取得
- CUI単発実行
- 並列: デフォルト2、最大10
- 都道府県全域検索でも目標10分以内
- 初期リリースの失敗ポリシーは停止（`stop`）固定
- 終了コード運用:
  - 正常終了 `0`
  - 入力不備 `2`
  - 取得失敗停止 `3`
  - その他例外 `1`

## 3. 技術選定（言語・ライブラリ・ミドルウェア）
### 3.1 採用方針
- 「個人用 CLI + 単発実行」のため、依存は最小限にする。
- DOM変化に備え、抽出ロジックは「構造化データ優先 + CSS抽出フォールバック」の二段構えにする。
- 速度要件のため、非同期I/O + セマフォ並列制御を採用する。

### 3.2 採用技術
| 区分 | 採用 | 用途 | 採用理由 |
|---|---|---|---|
| 言語 | Python 3.12+ | 実装言語 | 仕様前提、Playwright連携が安定 |
| 自動化 | Playwright for Python | ページ遷移/取得 | 仕様要件の中核 |
| 実行モデル | `asyncio`（標準） | 並列・待機制御 | 2〜10並列を軽量制御 |
| CLI | Typer | CUI引数/ヘルプ | 引数定義と保守性を両立 |
| バリデーション | Pydantic v2 | 入力/設定検証 | 終了コード2を明確化しやすい |
| HTML/XML解析 | selectolax + lxml | 宿情報抽出/area.xml解析 | HTMLはselectolax、`area.xml` はlxmlに責務分離。依存削減（lxml除外）はv1では見送り、安定動作確認後に再評価 |
| リトライ | Tenacity | 一時失敗時再試行 | 取得失敗率低減 |
| ログ | `logging`（標準） | 進行・失敗記録 | 依存を増やさず十分運用可能 |
| テスト | pytest + pytest-asyncio | 単体/結合検証 | 非同期処理の検証に必須 |

### 3.3 ミドルウェア方針
- 必須ミドルウェア: なし（Webサーバー/DB/キューは初期不要）
- 理由:
  - 本ツールは個人利用の単発CLIであり、常駐APIや分散処理基盤は不要。
  - データ永続化は当面ファイル出力（CSV/JSON）で十分。
- 拡張時の候補（将来）:
  - SQLite: 実行キャッシュ/再開機能
  - Redis + ワーカー: 並列分散が必要になった場合のみ

### 3.4 依存パッケージ導入定義（2026-02-21）
- 実行時依存（`requirements.txt`）:
  - `playwright`
  - `typer`
  - `pydantic`
  - `selectolax`
  - `lxml`
  - `tenacity`
- 開発依存（`requirements-dev.txt`）:
  - `pytest`
  - `pytest-asyncio`
- ブラウザ実行基盤:
  - `python -m playwright install chromium` を必須セットアップ手順にする。

## 4. アーキテクチャ設計
### 4.1 コンポーネント構成
- `cli`
  - 引数解釈、実行モード選択、終了コード返却
- `config`
  - タイムアウト、並列数、待機秒数、リトライ設定、固定設定値の管理
- `area_resolver`
  - 同梱 `area.xml`（固定スナップショット）から都道府県配下SMLコード展開
- `query_builder`
  - 検索URLテンプレ生成（宿泊日、人数、食事、こだわり条件）
- `crawler`
  - Playwrightセッション管理
  - ページネーション追跡（`idx` / `dispStartIndex`）
  - セマフォで並列上限制御
- `extractor`
  - 宿名/宿URL/プラン名/価格を抽出
  - 構造化データ優先、DOMセレクタはフォールバック
- `deduplicator`
  - URL正規化（クエリ除去、パス基準）で統合
- `output`
  - 結果JSONは `stdout`、ログ/進捗/エラーは `stderr` に分離して出力
- `error_policy`
  - 初期リリースでは停止のみを制御
  - 例外と終了コードを一元化

### 4.2 処理フロー
1. CLIで入力受領、Pydanticで検証。
2. 検証NGは終了コード `2` で終了。
3. 都道府県指定時は `area_resolver` でSMLに分解。
4. SMLまたは宿名単位で検索タスクを生成。
5. `crawler` が並列実行し、各タスクでページを巡回して `extractor` へ渡す。
6. `deduplicator` で宿単位統合。
7. 取得結果を出力し、失敗情報と件数をログへ記録。
8. 停止ポリシー（固定）に従い終了コード `0/1/3` を返却。

### 4.3 性能・信頼性設計
- 並列数: `2` デフォルト、`10` 上限をハード制約。
- レート制御:
  - エリア間待機は `1000ms`（v1の暫定安全値）
  - リトライは指数バックオフ（`1s -> 2s -> 4s`、jitterあり、上限`8s`）
- リトライ回数: 最大 `3` 回（初回 + 再試行2回）。
- タイムアウト分離:
  - ページロード timeout
  - 要素待機 timeout
- 中断判定:
  - アクセス集中/制限兆候の検出時は強制停止（コード `3`）。
- 失敗時挙動:
  - エリア単位失敗でもスキップせず停止（件数・URL・エリアをログ出力）。
  - `continue-on-error` は v1では採用しない（将来拡張）。

### 4.4 アクセス制限検出とリトライ優先順位（Q2確定）
- 方針:
  - `アクセス制限兆候 > Tenacityリトライ` を優先する。
  - 制限兆候が出たリクエストはリトライせず即停止（終了コード `3`）。
  - Tenacity は通信タイムアウト/一時的なネットワーク失敗/5xx（制限兆候なし）にのみ適用する。
- 初期キーワード集合（`restriction_keywords_v1`）:
  - `アクセスが集中`
  - `ただいまアクセスが集中`
  - `ご利用を制限`
  - `しばらく時間をおいて`
  - `短時間に多くのアクセス`
  - `不正なアクセス`
  - `Too Many Requests`
  - `Forbidden`
- 追加シグナル:
  - HTTPステータス `429` / `403`
- 判定条件（実装ルール）:
  - 条件A: HTTPステータスが `429` または `403` の場合は即 `AccessRestrictedError`。
  - 条件B: 本文テキストに `restriction_keywords_v1` のいずれかが含まれ、かつ宿一覧リンク（`/yad`）を1件も検出できない場合は即 `AccessRestrictedError`。
  - `AccessRestrictedError` は Tenacity の `retry` 対象外例外として扱う。
  - `4xx`（`429/403` を除く）は原則 `retry` 対象外とする。

### 4.5 `area.xml` 取得・更新方針（v1）
- 実行時に毎回リモート取得はしない。v1は同梱スナップショットを参照する。
- 通信失敗を実行時故障点にしないため、`area.xml` はアプリ資材として扱う。
- エリア定義更新が必要になった場合のみ、手動更新フローで差し替える。
